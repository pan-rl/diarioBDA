En la primera semana, revisamos el cronograma del curso y los conceptos fundamentales que se abordarían durante el cuatrimestre. El modelo objeto-relacional combina las ventajas de las bases de datos relacionales con la programación orientada a objetos, ofreciendo mayor flexibilidad y adaptabilidad.

Bases de Datos por Objetos: Postgres vs Oracle

IBM es considerado el pionero en el desarrollo de bases de datos en la industria. En el contexto actual, es importante entender cómo integrar múltiples fuentes de datos, donde una solución práctica es utilizar formatos planos como TXT como intermediario para el procesamiento y migración de datos.

🔍 Conceptos Clave de la Programación Orientada a Objetos en Bases de Datos

Características Principales
  • Modularidad: Permite estructurar el código en módulos independientes
  • Herencia: Las clases pueden heredar atributos y métodos de otras clases
  • Encapsulamiento: Ocultamiento de los detalles internos de implementación
  • Polimorfismo: Capacidad de procesar objetos de diferentes formas
Conceptos Avanzados
  • Composición: Relación fuerte donde el ciclo de vida de los componentes depende del contenedor
  • Agregación: Relación más flexible donde los objetos pueden ser compartidos
  • Tipos de Datos Definidos por el Usuario (UDT): Permiten crear estructuras de datos personalizadas

Modelado de Datos

El modelado en bases de datos orientadas a objetos implica crear representaciones que evitan información redundante. Un modelo de datos orientado a objetos se basa en relaciones de tipo "todo-parte" donde los objetos pueden contener o poseer otros objetos.

💡 Ventajas del Modelo Objeto-Relacional

  • Mayor capacidad de modelado de datos complejos
  • Mejor integración con lenguajes de programación orientados a objetos
  • Mayor flexibilidad en el diseño de la base de datos
  • Mejor rendimiento para ciertos tipos de consultas

Puntos clave:

  • Integración de conceptos de POO: encapsulamiento, herencia y polimorfismo
  • Superación de limitaciones del modelo relacional tradicional
  • Uso de identificadores únicos (OID) y tipos de datos personalizados
  • Creación de estructuras de datos más complejas y realistas

-- Crear una función que devuelve el inventario disponible
CREATE FUNCTION dbo.ufnGetInventoryStock(@ProductID int)
RETURNS int
AS
-- Retorna el nivel de inventario para el producto
BEGIN
    DECLARE @ret int;
    SELECT @ret = SUM(p.Quantity)
    FROM Production.ProductInventory p
    WHERE p.ProductID = @ProductID
        AND p.LocationID = '6';
     IF (@ret IS NULL)
        SET @ret = 0;
    RETURN @ret;
END;

FROM Production.Product
WHERE ProductModelID BETWEEN 75 AND 80;
GO

¿Qué es Apollo Server?

Apollo Server es un servidor de código abierto de GraphQL que funciona con cualquier base de datos y con cualquier cliente de GraphQL. Es compatible con todas las funciones principales de GraphQL y se integra perfectamente con otras herramientas del ecosistema GraphQL.

🔍 Puntos clave de Apollo Server:

  • Esquema fuertemente tipado: Define tu API de manera clara y precisa
  • Integración con múltiples fuentes de datos: Conecta con bases de datos, APIs REST, gRPC, etc.
  • Herramientas de desarrollo: Incluye Apollo Studio para probar y documentar tu API
  • Rendimiento optimizado: Incluye características como el almacenamiento en caché y la carga por lotes
  • Seguridad integrada: Validación de esquemas y protección contra consultas maliciosas

-- Cuerpo del tipo tp_Persona
CREATE OR REPLACE TYPE BODY tp_Persona AS
  MEMBER FUNCTION edad RETURN NUMBER IS
    a NUMBER;
    d DATE;
  BEGIN
    d:= CURRENT_DATE;
    a:= TO_CHAR(d,'YYYY') - TO_CHAR(fechanac,'YYYY');
    IF ((TO_CHAR(d,'MM') < TO_CHAR(fechanac,'MM')) OR 
       (TO_CHAR(d,'MM') < TO_CHAR(fechanac,'MM')) AND
       (TO_CHAR(d,'DD') < TO_CHAR(fechanac,'DD')))
    THEN
      a:= a+1;
    END IF;
    RETURN a;
  END;
  
  MEMBER FUNCTION nombre_completo RETURN VARCHAR2 IS
    salida VARCHAR2(50);
  BEGIN
    salida:= TRIM(apellp) || ' ' || TRIM(apellm) || ', ' || TRIM(nombre);
    RETURN salida;
  END;
  
  MAP MEMBER FUNCTION iden RETURN NUMBER IS
  BEGIN
    RETURN id;
  END;
END;

Semana 6 - Particionamiento de Bases de Datos

Optimización de Bases de Datos mediante Particionamiento

En esta sesión exploramos las técnicas de particionamiento de bases de datos, una estrategia fundamental para mejorar el rendimiento en entornos con grandes volúmenes de información.

Figura: Comparativa entre particionamiento vertical y horizontal

Conceptos Clave

El particionamiento permite dividir grandes conjuntos de datos en partes más manejables, mejorando significativamente el rendimiento de las consultas y facilitando el mantenimiento de las bases de datos.

🔍 Tipos de Particionamiento

Particionamiento Vertical
  • Divide una tabla en múltiples tablas más pequeñas
  • Mantiene una clave primaria común entre las tablas resultantes
  • Ideal para separar columnas de uso frecuente de las menos utilizadas
  • Reduce la carga de E/S al acceder solo a los campos necesarios
Particionamiento Horizontal
  • Divide los registros de una tabla según un criterio específico (ej: por año o región)
  • Cada partición contiene una copia completa de la estructura de la tabla original
  • SQL Server las gestiona como una única tabla lógica
  • Permite distribuir físicamente los datos en diferentes ubicaciones

Ejemplo de particionamiento
-- Crear función de partición
CREATE PARTITION FUNCTION pf_Anios (int)
AS RANGE LEFT FOR VALUES (2019, 2020, 2021, 2022, 2023);

-- Crear esquema de partición
CREATE PARTITION SCHEME ps_Anios

  • Facilita el mantenimiento al permitir operaciones en particiones individuales
  • Reduce los tiempos de respuesta al procesar menos datos por consulta
  • Permite estrategias de almacenamiento más eficientes

Semana 8 - Bases de Datos Distribuidas

Introducción a las Bases de Datos Distribuidas

Una base de datos distribuida es un sistema en el cual la información no se encuentra almacenada en un solo servidor central, sino que se reparte entre varios equipos (nodos) conectados en red. Cada nodo puede almacenar una parte de los datos o una copia completa, y todos trabajan de manera coordinada como si fueran una sola base unificada.

🔍 ¿Por qué surgen las bases de datos distribuidas?

Este modelo surge para mejorar el rendimiento, la disponibilidad y la tolerancia a fallos. Si un nodo falla, los demás pueden seguir funcionando y ofreciendo acceso a los datos, lo que asegura continuidad en los servicios.

Características principales

Transparencia de distribución

Los usuarios acceden a la base como si fuera única, aunque en realidad esté dividida en múltiples nodos.

Replicación de datos

Consiste en tener copias de la misma información en distintos servidores, garantizando disponibilidad si un nodo falla.

Fragmentación o particionamiento

División de la base en segmentos (por rangos de clientes, regiones o tipos de datos).

Alta disponibilidad

El sistema debe estar en funcionamiento el mayor tiempo posible, aun cuando haya fallos.

Escalabilidad

Posibilidad de añadir más nodos para manejar mayor cantidad de usuarios o datos.

✅ Ventajas

  • Mejor rendimiento al distribuir la carga de trabajo
  • Disponibilidad continua: si un servidor cae, otro puede responder
  • Escalabilidad flexible según la demanda
  • Cercanía a los usuarios: datos almacenados geográficamente cercanos

⚠️ Desafíos

  • Mantener la coherencia de los datos en todos los nodos
  • Mayor complejidad en la administración
  • Seguridad y control de acceso distribuidos
  • Sincronización de datos en tiempo real

1. Arquitectura de bases de datos distribuidas

Una base de datos distribuida es aquella en la que los datos se almacenan en múltiples nodos conectados por una red, pero se gestionan de manera que el usuario percibe el sistema como una sola base unificada.

🔍 Características clave

  • Transparencia: El usuario no debe notar dónde están los datos (transparencia de localización y replicación)
  • Ventajas: Mayor disponibilidad, tolerancia a fallos y mejor tiempo de respuesta
  • Desafíos: Sincronización de datos, seguridad y consistencia global

2. Paralelismo en bases de datos

El paralelismo busca ejecutar múltiples operaciones en simultáneo para mejorar el rendimiento. Se puede dar en varios niveles:

Paralelismo de consultas

Ejecución de una consulta en diferentes nodos para reducir el tiempo de respuesta.

Paralelismo de transacciones

Varias transacciones pueden ejecutarse a la vez, siempre que no entren en conflicto.

Paralelismo de datos

Fragmentación de tablas y procesamiento en paralelo de diferentes fragmentos.

Esto es clave en sistemas modernos como Google BigQuery o Amazon Redshift, que procesan millones de registros en segundos.

3. Confiabilidad (Reliability)

En una BDD, la confiabilidad es crítica. Se refiere a la capacidad del sistema para operar correctamente pese a fallos de hardware, software o red.

Tolerancia a fallos

Mediante replicación de datos y protocolos de consenso (ej. Paxos, Raft).

Durabilidad

Uso de registros de transacciones (logs) para garantizar que las operaciones confirmadas no se pierdan.

Disponibilidad

Sistemas redundantes y balanceadores de carga aseguran el acceso continuo a la información.

4. Modelo de Schroeder

Propuesto en el contexto de seguridad informática, el modelo de Schroeder enfatiza principios de diseño seguro para sistemas:

  • Economía de mecanismos: Mantener la arquitectura simple para evitar vulnerabilidades.
  • Defensa en profundidad: Múltiples capas de seguridad.
  • Privilegio mínimo: Cada usuario o proceso debe tener solo los permisos estrictamente necesarios.
  • Auditoría y control: Registro de eventos para detectar anomalías.

Estos principios aplicados a BDD refuerzan la seguridad y confiabilidad del sistema.

5. Share-Nothing Architecture

La arquitectura share-nothing es un modelo en el que cada nodo del sistema tiene sus propios recursos (CPU, memoria, almacenamiento) y no los comparte con otros nodos.

Escalabilidad horizontal

Se pueden agregar nodos sin grandes modificaciones.

Reducción de cuellos de botella

No existe competencia por recursos compartidos.

Aislamiento de fallos

Un nodo que falla no afecta directamente a los demás.

Ejemplo: Sistemas como Hadoop o Cassandra implementan este enfoque, siendo altamente escalables y resilientes.

Semana 11 - Búsqueda de Texto Completo

Full Text Search en SQL Server

La búsqueda de texto completo (Full Text Search) en Microsoft SQL Server permite realizar consultas avanzadas sobre datos almacenados en columnas de tipo texto. Para usarla es necesario crear un índice de texto completo, que puede incluir columnas de tipos como varchar, nvarchar, text, xml, entre otros.

🔍 Características clave de Full Text Search

  • Término simple: Búsqueda de una palabra o frase exacta
  • Término de prefijo: Búsqueda de palabras que comienzan con un texto dado
  • Término de inflexión: Búsqueda de formas gramaticales de una palabra
  • Término de proximidad (NEAR): Busca palabras cercanas entre sí
  • Términos sinónimos y ponderados: Mediante diccionarios o valores de relevancia

Las consultas de texto completo se realizan con predicados y funciones de Transact-SQL como CONTAINS, FREETEXT, CONTAINSTABLE y FREETEXTTABLE.

Ejemplo de consulta con CONTAINS:
-- Buscar productos que contengan 'cámara' en su descripción
SELECT * FROM Productos 
WHERE CONTAINS(descripcion, 'cámara')

-- Búsqueda con términos de proximidad
SELECT * FROM Documentos
WHERE CONTAINS(contenido, 'NEAR((base, datos), 5, TRUE)')

Semana 13 - Vistas Materializadas

Optimización con Vistas Materializadas

Las vistas materializadas son objetos de base de datos que almacenan el resultado de una consulta como una tabla física. A diferencia de las vistas estándar, las vistas materializadas almacenan los datos físicamente y se pueden actualizar periódicamente.

Ventajas
  • Mejor rendimiento en consultas complejas
  • Reducción de la carga en el sistema
  • Útiles para datos históricos o resumidos
Consideraciones
  • Ocupan espacio de almacenamiento
  • Pueden quedar desactualizadas
  • Diferentes sintaxis según el SGBD
Ejemplo en SQL Server:
-- Crear una vista materializada (SQL Server usa Índices Clusterizados en Vistas)
CREATE VIEW dbo.VentasMensuales
WITH SCHEMABINDING
AS
    SELECT 
        YEAR(FechaVenta) AS Año,
        MONTH(FechaVenta) AS Mes,
        COUNT_BIG(*) AS TotalVentas,
        SUM(Total) AS MontoTotal
    FROM dbo.Ventas
    GROUP BY YEAR(FechaVenta), MONTH(FechaVenta);
GO

-- Crear índice clusterizado para materializar la vista
CREATE UNIQUE CLUSTERED INDEX IX_VentasMensuales 
ON dbo.VentasMensuales(Año, Mes);

� Conceptos Clave de Vistas Materializadas

  • Almacenamiento físico: Los datos se almacenan en disco
  • Actualización periódica: Se pueden actualizar según una programación
  • Mejora del rendimiento: Reducen la carga de consultas complejas
  • Útiles para datos históricos: Resumen de datos a lo largo del tiempo
